# Правило: написание нового кода с учетом архитектуры FSD

При написании любого нового кода необходимо соблюдать архитектуру Feature-Sliced Design (FSD).

## Основной принцип

Код должен размещаться в соответствии с его уровнем ответственности, а не по техническому типу файла. Перед созданием нового файла или модуля нужно определить, к какому слою FSD относится логика.

## Слои FSD

Используй слои в следующем порядке:

- `app` — инициализация приложения, провайдеры, роутинг, глобальные стили, конфигурация приложения.
- `pages` — страницы приложения и их композиция.
- `widgets` — крупные самостоятельные UI-блоки, собранные из features, entities и shared.
- `features` — пользовательские сценарии и действия.
- `entities` — бизнес-сущности предметной области.
- `shared` — переиспользуемый код без бизнес-смысла: UI-компоненты, утилиты, API-клиент, конфиги, типы.

## Правило размещения нового кода

Перед написанием кода агент должен ответить на вопрос: “Какую ответственность несет этот код?”

- Если код относится к конкретному пользовательскому действию — помещай его в `features`.
- Если код описывает бизнес-сущность — помещай его в `entities`.
- Если код является частью страницы — помещай его в `pages`.
- Если код является крупным блоком интерфейса — помещай его в `widgets`.
- Если код универсальный и не содержит бизнес-логики — помещай его в `shared`.
- Если код отвечает за настройку всего приложения — помещай его в `app`.

## Правила импортов

Нельзя импортировать код из вышележащих слоев в нижележащие.

Разрешенное направление зависимостей:

```txt
app → pages → widgets → features → entities → shared
```

Каждый слой может импортировать только нижележащие слои.

Примеры:

- `features` может импортировать из `entities` и `shared`.
- `entities` может импортировать только из `shared`.
- `shared` не должен импортировать из `entities`, `features`, `widgets`, `pages` или `app`.

## Public API

Каждый слайс должен экспортировать внешний API через `index.ts`.

Импортировать внутренние файлы другого слайса напрямую запрещено.

Нельзя:

```ts
import { UserCard } from '@/entities/user/ui/UserCard'
```

Нужно:

```ts
import { UserCard } from '@/entities/user'
```

## Структура слайса

Внутри слайса код нужно группировать по сегментам:

```txt
model/
ui/
api/
lib/
config/
types/
```

Используй только нужные сегменты. Не создавай пустые директории.

## Правило для нового кода

При создании нового кода агент обязан:

1. Определить слой FSD.
2. Определить слайс, если слой предполагает слайсы.
3. Разместить код в подходящем сегменте.
4. Проверить направление импортов.
5. Использовать Public API.
6. Не смешивать бизнес-логику, UI и инфраструктурный код без необходимости.
7. Не выносить код в `shared`, если он связан с конкретной бизнес-сущностью или фичей.

## Запреты

Запрещено:

- помещать бизнес-логику в `shared`;
- импортировать из внутренних файлов чужого слайса;
- нарушать направление зависимостей;
- создавать универсальные абстракции заранее без реального повторного использования;
- смешивать несколько независимых фич в одном слайсе;
- размещать код по принципу “куда проще”, игнорируя ответственность модуля.

## Если слой неочевиден

Если непонятно, куда поместить код, агент должен выбрать самый низкий подходящий слой, но не `shared` по умолчанию.

`shared` используется только для кода, который не знает ничего о бизнесе приложения.
